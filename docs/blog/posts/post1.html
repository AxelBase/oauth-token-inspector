Introducing OAuth Token Inspector

Published: November 10, 2025

In an era where data privacy is paramount, developers need tools they can trust. The OAuth Token Inspector is a fully client-side, open-source web utility designed to decode and inspect JSON Web Tokens (JWTs) without ever transmitting sensitive data to external servers. Built with SvelteKit, styled using Bootstrap 5.3.3, and deployed via GitHub Pages, this tool ensures your OAuth access and refresh tokens remain secure during debugging and integration testing.

Why a Client-Side JWT Inspector?

Traditional online JWT decoders require you to paste tokens into third-party websites — a risky practice that exposes authentication secrets. Even "trusted" services may log inputs, retain data, or be compromised. The OAuth Token Inspector eliminates this risk by performing all decoding and validation directly in your browser. No network requests are made beyond loading the static page and Bootstrap CDN assets.

Whether you're integrating with Google, Auth0, Azure AD, or a custom OAuth provider, this tool helps you understand token structure, validate claims, and debug expiration issues — all while maintaining full control over your data.

Core Features at a Glance

  • Instant Decoding: Paste any JWT and see Header, Payload, and Signature instantly.
  • Expiration Validation: Color-coded status for exp claim — Valid (green), Expired (red), or No exp claim (yellow).
  • Standard Claim Tooltips: Hover over iss, sub, aud, etc., for RFC 7519 definitions.
  • Copy JSON: One-click export of decoded header and payload for API testing.
  • Dark Mode: Toggle between light and dark themes using Bootstrap’s data-bs-theme.

How It Works Under the Hood

The tool uses vanilla JavaScript to implement Base64URL decoding and JSON parsing. The jwt.js utility splits the token into three parts, decodes the first two, and displays the raw signature. No external libraries are used for token processing — keeping the bundle lightweight and auditable.

All functionality aligns with the Software Requirements Specification (SRS), including 10 functional requirements (FR-01 to FR-10) and 7 non-functional requirements (NFR-01 to NFR-07). The application is stateless, responsive, and accessible (WCAG 2.1 AA compliant).

FAQ

Is it safe to paste production tokens?

Yes — nothing leaves your browser. However, avoid pasting tokens with PII. Use test or sandbox tokens when possible.

Does it verify signatures?

No. It displays the raw signature for manual verification. Use backend libraries like jsonwebtoken for cryptographic validation.

Can I host my own version?

Absolutely. Fork the GitHub repo and deploy in minutes using GitHub Pages.
